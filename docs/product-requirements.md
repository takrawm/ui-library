# プロダクト要求定義書

## 1. プロダクトビジョンと目的

### ビジョン

shadcn/ui をベースとした、再利用可能な UI コンポーネントライブラリを構築・蓄積し、Web システム（デスクトップ画面）の開発を加速させる。

### 目的

- 高品質な UI コンポーネントを体系的に蓄積し、プロジェクト横断で再利用可能にする
- shadcn/ui の設計思想（合成パターン、アクセシビリティ、カスタマイズ性）を踏襲しつつ、より高レベルなコンポーネントを提供する
- コンポーネントカタログとして、各コンポーネントの仕様・使い方・設計意図を明文化する

## 2. ターゲットユーザーと課題・ニーズ

### ターゲットユーザー

- **主要ユーザー**: 自身（および将来的にチームメンバー）が Web アプリケーションを開発する際にコンポーネントを利用する開発者
- **二次ユーザー**: コンポーネントカタログを参照し、UI の一貫性を確認するデザイナー・レビュアー

### 課題

- プロジェクトごとに似たような UI コンポーネントをゼロから作り直すことによる非効率
- コンポーネントの設計意図や使い方が属人化し、再利用・引き継ぎが困難
- アクセシビリティやキーボード操作対応が後回しになりやすい

### ニーズ

- すぐに使える高品質なコンポーネント群
- 各コンポーネントの設計意図・使い方が明文化されたカタログ
- shadcn/ui との互換性を維持しつつ、より実用的なレベルまで組み上げたコンポーネント

## 3. 主要な機能一覧

### プレイグラウンド

コンポーネントの動作確認・組み合わせデモを Next.js ベースのプレイグラウンドで提供する。

| 機能 | 説明 |
|---|---|
| コンポーネントショーケース | 各 shadcn/ui ベースコンポーネントの全バリエーション表示 |
| 組み合わせデモ | 複数コンポーネントを組み合わせた実用的な UI パターン |
| コンポーネントブラウザ | 利用可能なコンポーネントの一覧・インポートパス参照 |
| ファイルシステム連動 | `playground/components/` `playground/demos/` 配下を自動走査して一覧表示 |

### コンポーネントカテゴリ

| カテゴリ | 説明 | 例 |
|---|---|---|
| layouts | ページ全体の構造 | サイドバー付きレイアウト、ダッシュボードシェル |
| navigation | ナビゲーション系 | サイドバー、ブレッドクラム、タブ |
| data-display | データ表示系 | カード、テーブル、リスト、統計表示 |
| data-input | データ入力系 | フォーム、フィルター、検索バー |
| feedback | フィードバック系 | トースト、モーダル、プログレスバー |
| interactive | インタラクティブ系 | ドラッグ&ドロップ、ソート可能リスト |
| ui | shadcn/ui ベース | shadcn add で追加されるベースコンポーネント |

### コンポーネント提供形態

- メインの `.tsx` 実装ファイル
- テストファイル（`.test.tsx`）
- 型定義ファイル（`types.ts`、必要な場合）
- JSDoc コメントによる設計意図・使い方の記載
- Storybook 形式の使用例（`.stories.tsx`）（将来導入予定）
- README（設計意図・使い方・制約）（成熟したコンポーネントに対して作成）

### コンポーネントカタログ

- Next.js App Router によるプレイグラウンド（`/playground`）
  - コンポーネントショーケース（`/playground/components/[name]`）
  - 組み合わせデモ（`/playground/demos/[slug]`）
- 全コンポーネントの一覧と状態を管理する `CATALOG.md`（補助的な参照用）

## 4. 成功の定義

- コンポーネントが体系的に分類・蓄積され、カタログから検索・参照できる
- 各コンポーネントが props なしでもデフォルト表示でき、即座に動作確認可能
- アクセシビリティ（キーボード操作、スクリーンリーダー対応）が担保されている
- `pnpm build` および `pnpm lint` がエラーなく通る状態が常に維持されている
- コンポーネントの設計意図と使い方が README に明文化されている

## 5. ビジネス要件

- コンポーネントは段階的に蓄積する（一度にすべてを作らない）
- shadcn/ui の更新に追従可能な構造を維持する（`ui/` 内のファイルは直接編集しない）
- デスクトップファーストで設計し、最低限タブレットサイズまで対応する

## 6. ユーザーストーリー

### US-1: コンポーネントの追加

開発者として、新しい UI コンポーネントを定義されたカテゴリ・構成ルールに従って追加できる。これにより、ライブラリが体系的に成長する。

### US-2: コンポーネントの利用

開発者として、カタログからコンポーネントを見つけ、README の使い方に従ってプロジェクトに組み込める。これにより、ゼロから作り直す手間が省ける。

### US-3: コンポーネントの動作確認

開発者として、プレイグラウンド（`/playground`）でコンポーネントの見た目と動作を確認できる。これにより、実装前に期待通りの UI かどうか判断できる。

### US-5: 組み合わせデモの生成

開発者として、Claude Code または Cursor に要件を伝えて、複数コンポーネントを組み合わせたデモページを `playground/demos/` に自動生成できる。これにより、コンポーネントの組み合わせパターンを素早く試せる。

### US-4: shadcn/ui ベースコンポーネントの管理

開発者として、shadcn/ui の MCP サーバーを使って最新仕様を確認し、ベースコンポーネントを追加・更新できる。これにより、常に最新の shadcn/ui に基づいた開発が可能になる。

## 7. 受け入れ条件

### AC-1: コンポーネント構成

- 各カスタムコンポーネントが所定のディレクトリ構成（`.tsx`, `.test.tsx`, `types.ts`）で作成されている（`.stories.tsx`, `README.md` は段階的に追加）
- コンポーネントが適切なカテゴリに配置されている

### AC-2: 品質基準

- `pnpm build` がエラーなく通る
- `pnpm lint` がエラーなく通る
- TypeScript のエラーが 0 件
- props なしでもデフォルト表示が壊れない

### AC-3: アクセシビリティ

- Tab キーでフォーカス可能な要素に到達できる
- Enter / Space キーで操作可能
- 適切な `aria-*` 属性が設定されている

### AC-4: ドキュメント

- 各コンポーネントに JSDoc コメントで概要・主要 Props の説明が記載されている（成熟コンポーネントは README も作成）
- 新しいコンポーネントがカタログページに反映されている（`CATALOG.md` は補助的な参照用）

## 8. 機能要件

### FR-1: コンポーネントライブラリ基盤

- Next.js (App Router) + TypeScript によるプロジェクト構成
- shadcn/ui のベースコンポーネントを `src/components/ui/` に配置
- カスタムコンポーネントをカテゴリ別ディレクトリに配置

### FR-2: コンポーネントカタログ（プレイグラウンド）

- `/`（`src/app/page.tsx`）は `/playground` にリダイレクト
- プレイグラウンドトップ（`src/app/playground/page.tsx`）でショーケース + デモ一覧を表示
- コンポーネントショーケース: `playground/components/[name]/page.tsx` で各コンポーネントの全バリエーション表示
- 組み合わせデモ: `playground/demos/[slug]/page.tsx` で複数コンポーネントの組み合わせ
- ファイルシステム走査による自動一覧生成（Server Component）

### FR-5: プレイグラウンドのデモ生成

- Claude Code / Cursor でプロンプトを入力し、`playground/demos/` 配下にデモページを自動生成
- サンプルデータは各ページファイル内にハードコード（DB 不要）
- Next.js ホットリロードにより、ファイル生成後即座にプレビュー可能

### FR-3: shadcn/ui MCP 連携

- MCP サーバー経由で shadcn/ui の最新仕様を確認可能
- `list_components` / `get_component` / `get_block` を活用した開発フロー

### FR-4: コンポーネント設計原則の遵守

- 合成パターン（Composition）による設計
- Controlled / Uncontrolled 両対応
- ジェネリクスを活用した型安全な props 設計

## 9. 非機能要件

### NFR-1: パフォーマンス

- コンポーネントのバンドルサイズを最小限に保つ
- 不要な再レンダリングを防ぐ（`React.memo`, `useCallback` の適切な使用）

### NFR-2: 保守性

- shadcn/ui のアップデートに追従可能な構造
- `ui/` 内のベースコンポーネントは直接編集しない
- 各コンポーネントの設計意図が README で明文化されている

### NFR-3: 開発体験

- TypeScript 厳密モード（`strict: true`）
- ESLint / Prettier による一貫したコードスタイル
- テストフレームワーク: Vitest + React Testing Library + vitest-axe
- Storybook による開発中のプレビュー（将来導入予定、初期は Next.js カタログページで代替）

### NFR-4: アクセシビリティ

- WCAG 2.1 AA レベルを目標
- キーボード操作の完全対応
- スクリーンリーダー対応（適切な aria 属性）

### NFR-5: ブラウザ対応

- Chrome（最新版）
- Firefox（最新版）
- Safari（最新版）
- Edge（最新版）
