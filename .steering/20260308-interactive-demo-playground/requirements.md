# 要求内容：インタラクティブデモプレイグラウンド

## 概要

UI コンポーネントライブラリに、自然言語プロンプトからコンポーネントデモを生成・操作・保存できるインタラクティブなプレイグラウンド機能を追加する。

## 背景と目的

現在のコンポーネントカタログ（`src/app/page.tsx`）は静的な一覧表示に留まっており、コンポーネントの組み合わせや動的な振る舞い（クリック、ドラッグ&ドロップ、フォーム入力など）を試すには、コードを書いて `pnpm dev` で確認する必要がある。

本機能により、以下を実現する：

1. **自然言語でデモを生成**: プロンプトを入力するだけで、ライブラリ内のコンポーネントを組み合わせたデモ UI を自動生成する
2. **会話的にデモを改善**: 生成されたデモに対して、追加の自然言語プロンプトで修正・改善を指示できる
3. **インタラクティブな動作確認**: 生成されたデモ上でクリック、ドラッグ&ドロップ、キーボード操作など、実際の動作を確認できる
4. **デモの永続化**: 作成したデモをデータベースに保存し、後から再利用できる

## 機能要件

### FR-P1: プレイグラウンド UI

Web アプリケーション上にプレイグラウンド画面を提供する。

- **プロンプト入力エリア**: 自然言語でデモの要件を入力できるテキストエリア
  - 例: 「3つのカードをドラッグ&ドロップで並べ替えられるリストを表示して」
  - 例: 「ログインフォームをバリデーション付きで表示して」
  - 例: 「統計カード4枚とデータテーブルのダッシュボードを表示して」
- **デモプレビューエリア**: 生成されたコンポーネントがレンダリングされ、実際に操作可能なエリア
  - クリック、ホバー、フォーカス、ドラッグ&ドロップなどのインタラクションが動作する
  - レスポンシブ表示の確認が可能（ビューポートサイズの切り替え）
- **コード表示エリア**（オプション）: 生成されたコードを確認・編集できるエリア
- **会話履歴エリア**: プロンプトとその結果の履歴を表示し、文脈を持った対話を可能にする

### FR-P2: デモ生成エンジン

自然言語プロンプトからコンポーネントデモを生成する。

- ライブラリ内の既存コンポーネント（`src/components/` 配下）をベースに組み合わせる
- shadcn/ui のベースコンポーネント（`src/components/ui/`）も利用可能
- 生成結果は React（TSX）コンポーネントとして出力される
- 生成されたコンポーネントはプレビューエリアで即座にレンダリングされる
- **LLM**: Claude API（Anthropic）を使用 — Claude Code / Cursor と同じモデルファミリー

### FR-P3: 会話的なデモ改善

生成済みのデモに対して、追加プロンプトで修正・改善を指示できる。

- 現在表示中のデモに対して、追加の自然言語プロンプトを送信できる
  - 例: 「カードの色を青系に変更して」
  - 例: 「テーブルにソート機能を追加して」
  - 例: 「レイアウトを2カラムから3カラムに変えて」
- 会話の文脈（これまでのプロンプトと生成コード）を保持し、差分的な修正が可能
- 修正のたびにプレビューが即座に更新される
- 会話をリセットして新しいデモを最初から作り直すこともできる

### FR-P4: デモの保存と管理

作成したデモを SQLite データベースに保存し、管理する。

- デモの保存（タイトル、会話履歴、最終生成コード、作成日時）
- 保存済みデモの一覧表示
- 保存済みデモの読み込み・再表示（会話履歴も含めて復元）
- 保存済みデモの削除
- 保存済みデモからの会話続行（読み込んだ後にさらに修正を指示）

### FR-P5: コンポーネント参照

プレイグラウンドからライブラリ内のコンポーネントを参照できる。

- 利用可能なコンポーネントの一覧表示
- 各コンポーネントの props 情報の表示
- コンポーネントの使用例の表示

## 非機能要件

### NFR-P1: 技術スタック

既存の技術スタックを踏襲する。

- **フレームワーク**: Next.js (App Router) — 既存プロジェクトの一部として構築
- **UI**: shadcn/ui ベースコンポーネント + ライブラリ内カスタムコンポーネント
- **言語**: TypeScript + React（TSX）
- **データベース**: SQLite + Prisma ORM（または Drizzle ORM）
  - ファイルベースでセットアップが簡単、個人利用に最適
  - devcontainer / WSL 環境でも追加設定なしで動作
- **LLM**: Claude API（Anthropic）
  - Claude Code / Cursor で使用しているのと同じ Claude モデルファミリーを API 経由で呼び出す
  - Next.js Server Actions または API Routes からサーバーサイドで呼び出す（API キーをクライアントに露出させない）
  - ストリーミングレスポンス対応で生成過程をリアルタイム表示

### NFR-P2: パフォーマンス

- デモ生成のレスポンス時間は実用的な範囲内とする（Claude API の応答速度に依存）
- ストリーミングにより生成過程を逐次表示し、体感待ち時間を短縮する
- 生成済みデモの読み込みは即座に行われること
- プレビューエリアでの操作はスムーズであること（フレームレート低下なし）

### NFR-P3: セキュリティとコード実行

個人利用を前提とした実用的なレベルのセキュリティを確保する。

- **コード実行方式**: Sandpack（CodeSandbox 提供のブラウザ内サンドボックス）を推奨
  - iframe 内で生成コードを実行するため、メインアプリとは分離される
  - ライブラリ内のコンポーネントを依存関係として提供可能
  - eval ベースの直接実行は避ける（個人利用であっても XSS リスクを不必要に負わない）
- **DB 入力バリデーション**: Prisma / Drizzle の型安全なクエリで基本的なインジェクション対策
- **API キー管理**: `.env.local` で管理し、サーバーサイドのみで使用

## ユーザーストーリー

### US-P1: デモの自然言語生成

開発者として、プレイグラウンドに自然言語で「統計カードを4枚並べたダッシュボードのヘッダー部分を表示して」と入力すると、ライブラリ内のコンポーネントを使ったデモが自動生成される。これにより、コードを書かずにコンポーネントの組み合わせ方を素早く検証できる。

### US-P2: 会話的なデモ改善

開発者として、生成されたデモを見ながら「カードの間隔をもう少し広げて」「ヘッダーにアイコンを追加して」と追加指示を出すと、既存のデモが修正される。これにより、試行錯誤しながら理想のレイアウトに近づけることができる。

### US-P3: インタラクティブな動作確認

開発者として、生成されたデモ上でドラッグ&ドロップやボタンクリックなどの操作を実際に試すことができる。これにより、コンポーネントの動的な振る舞いをコードを書く前に確認できる。

### US-P4: デモの保存と再利用

開発者として、良いデモが生成されたら会話履歴ごと保存し、後から再度読み込んで参照・追加修正できる。これにより、コンポーネントの使い方のサンプル集を蓄積できる。

## 受け入れ条件

### AC-P1: プレイグラウンド UI

- [ ] プレイグラウンドページ（`/playground` または類似パス）がアクセス可能
- [ ] テキストエリアに自然言語プロンプトを入力し、送信できる
- [ ] プレビューエリアにデモがレンダリングされる
- [ ] デモ上でインタラクション（クリック、ドラッグ&ドロップ等）が動作する
- [ ] 会話履歴が表示され、過去のプロンプトと結果を確認できる

### AC-P2: デモ生成と改善

- [ ] ライブラリ内のコンポーネントを使ったデモが生成される
- [ ] 生成結果が React コンポーネントとしてプレビューに表示される
- [ ] 生成済みデモに対して追加プロンプトで修正を指示できる
- [ ] 修正後のプレビューが即座に更新される
- [ ] 会話をリセットして新しいデモを開始できる
- [ ] 不正なプロンプトに対してエラーメッセージが表示される

### AC-P3: データ永続化

- [ ] デモを会話履歴ごと保存できる
- [ ] 保存済みデモの一覧を表示できる
- [ ] 保存済みデモを読み込んで再表示できる（会話履歴も復元される）
- [ ] 保存済みデモを削除できる
- [ ] 読み込んだデモに対して追加の修正を続行できる

### AC-P4: 品質基準

- [ ] `pnpm build` がエラーなく通る
- [ ] `pnpm lint` がエラーなく通る
- [ ] TypeScript のエラーが 0 件
- [ ] 生成コードの実行が Sandpack（iframe サンドボックス）内で行われる

## 決定済み事項

| 項目 | 決定内容 | 理由 |
|---|---|---|
| LLM | Claude API（Anthropic） | Claude Code / Cursor と同じモデル。開発者が慣れ親しんだモデルで一貫性がある |
| LLM 呼び出し方式 | サーバーサイド（Server Actions / API Routes） | API キーをクライアントに露出させない |
| データベース | SQLite | ファイルベースでセットアップ簡単、個人利用に最適 |
| コード実行方式 | Sandpack（iframe サンドボックス） | 個人利用でも eval は避けるべき。Sandpack はライブラリ依存関係の提供も可能 |

## 検討事項（design.md で詳細化）

### 1. ORM 選定

- **候補 A**: Prisma — エコシステムが大きく、型安全。マイグレーション管理が強力
- **候補 B**: Drizzle — 軽量で高速。SQLite との親和性が高い。バンドルサイズが小さい
- **検討ポイント**: 個人利用のシンプルなスキーマには Drizzle の方が軽量で適切か

### 2. 会話コンテキストの管理

- LLM に送る会話履歴のトークン数制限への対処
- 長い会話での要約・圧縮戦略
- システムプロンプトへのコンポーネント情報の組み込み方

### 3. Sandpack のカスタマイズ

- ライブラリ内コンポーネントを Sandpack の依存関係として提供する方法
- Tailwind CSS / shadcn/ui テーマの Sandpack 内での適用方法
- @dnd-kit 等の外部ライブラリの Sandpack 内での利用可否

### 4. 生成コードの品質

- 存在しないコンポーネントを参照した場合のエラーハンドリング
- 生成コードが構文エラーを含む場合のフォールバック表示
- LLM のシステムプロンプトで利用可能なコンポーネント一覧を提供する方式

## 制約事項

- 既存のコンポーネントカタログ機能（`src/app/page.tsx`）との共存を維持する
- 既存のコンポーネント設計原則（合成パターン、アクセシビリティ等）を崩さない
- プレイグラウンド機能の追加により、既存の `pnpm build` / `pnpm lint` が壊れないこと
- 個人利用を前提とするため、認証機能やマルチテナント対応は不要
