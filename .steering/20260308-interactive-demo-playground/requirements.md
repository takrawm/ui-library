# 要求内容：インタラクティブデモプレイグラウンド

## 概要

UI コンポーネントライブラリに、自然言語プロンプトからコンポーネントデモを生成・操作・保存できるインタラクティブなプレイグラウンド機能を追加する。

## 背景と目的

現在のコンポーネントカタログ（`src/app/page.tsx`）は静的な一覧表示に留まっており、コンポーネントの組み合わせや動的な振る舞い（クリック、ドラッグ&ドロップ、フォーム入力など）を試すには、コードを書いて `pnpm dev` で確認する必要がある。

本機能により、以下を実現する：

1. **自然言語でデモを生成**: プロンプトを入力するだけで、ライブラリ内のコンポーネントを組み合わせたデモ UI を自動生成する
2. **インタラクティブな動作確認**: 生成されたデモ上でクリック、ドラッグ&ドロップ、キーボード操作など、実際の動作を確認できる
3. **デモの永続化**: 作成したデモをデータベースに保存し、後から再利用・共有できる

## 機能要件

### FR-P1: プレイグラウンド UI

Web アプリケーション上にプレイグラウンド画面を提供する。

- **プロンプト入力エリア**: 自然言語でデモの要件を入力できるテキストエリア
  - 例: 「3つのカードをドラッグ&ドロップで並べ替えられるリストを表示して」
  - 例: 「ログインフォームをバリデーション付きで表示して」
  - 例: 「統計カード4枚とデータテーブルのダッシュボードを表示して」
- **デモプレビューエリア**: 生成されたコンポーネントがレンダリングされ、実際に操作可能なエリア
  - クリック、ホバー、フォーカス、ドラッグ&ドロップなどのインタラクションが動作する
  - レスポンシブ表示の確認が可能（ビューポートサイズの切り替え）
- **コード表示エリア**（オプション）: 生成されたコードを確認・編集できるエリア

### FR-P2: デモ生成エンジン

自然言語プロンプトからコンポーネントデモを生成する。

- ライブラリ内の既存コンポーネント（`src/components/` 配下）をベースに組み合わせる
- shadcn/ui のベースコンポーネント（`src/components/ui/`）も利用可能
- 生成結果は React（TSX）コンポーネントとして出力される
- 生成されたコンポーネントはプレビューエリアで即座にレンダリングされる

### FR-P3: デモの保存と管理

作成したデモをデータベースに保存し、管理する。

- デモの保存（タイトル、プロンプト、生成コード、作成日時）
- 保存済みデモの一覧表示
- 保存済みデモの読み込み・再表示
- 保存済みデモの削除
- 保存済みデモの編集（プロンプトの修正と再生成）

### FR-P4: コンポーネント参照

プレイグラウンドからライブラリ内のコンポーネントを参照できる。

- 利用可能なコンポーネントの一覧表示
- 各コンポーネントの props 情報の表示
- コンポーネントの使用例の表示

## 非機能要件

### NFR-P1: 技術スタック

既存の技術スタックを踏襲する。

- **フレームワーク**: Next.js (App Router) — 既存プロジェクトの一部として構築
- **UI**: shadcn/ui ベースコンポーネント + ライブラリ内カスタムコンポーネント
- **言語**: TypeScript + React（TSX）
- **データベース**: 要検討（候補: SQLite / PostgreSQL / Prisma ORM）
- **AI/LLM 連携**: 要検討（デモ生成に使用する LLM の選定と接続方式）

### NFR-P2: パフォーマンス

- デモ生成のレスポンス時間は実用的な範囲内とする（LLM の応答速度に依存）
- 生成済みデモの読み込みは即座に行われること
- プレビューエリアでの操作はスムーズであること（フレームレート低下なし）

### NFR-P3: セキュリティ

- 生成されたコードの実行にサンドボックスを適用する（XSS 等の防止）
- データベースへの入力にはバリデーションを適用する
- LLM API キーの安全な管理

## ユーザーストーリー

### US-P1: デモの自然言語生成

開発者として、プレイグラウンドに自然言語で「統計カードを4枚並べたダッシュボードのヘッダー部分を表示して」と入力すると、ライブラリ内のコンポーネントを使ったデモが自動生成される。これにより、コードを書かずにコンポーネントの組み合わせ方を素早く検証できる。

### US-P2: インタラクティブな動作確認

開発者として、生成されたデモ上でドラッグ&ドロップやボタンクリックなどの操作を実際に試すことができる。これにより、コンポーネントの動的な振る舞いをコードを書く前に確認できる。

### US-P3: デモの保存と再利用

開発者として、良いデモが生成されたら保存し、後から再度読み込んで参照・修正できる。これにより、コンポーネントの使い方のサンプル集を蓄積できる。

## 受け入れ条件

### AC-P1: プレイグラウンド UI

- [ ] プレイグラウンドページ（`/playground` または類似パス）がアクセス可能
- [ ] テキストエリアに自然言語プロンプトを入力し、送信できる
- [ ] プレビューエリアにデモがレンダリングされる
- [ ] デモ上でインタラクション（クリック、ドラッグ&ドロップ等）が動作する

### AC-P2: デモ生成

- [ ] ライブラリ内のコンポーネントを使ったデモが生成される
- [ ] 生成結果が React コンポーネントとしてプレビューに表示される
- [ ] 不正なプロンプトに対してエラーメッセージが表示される

### AC-P3: データ永続化

- [ ] デモを保存できる
- [ ] 保存済みデモの一覧を表示できる
- [ ] 保存済みデモを読み込んで再表示できる
- [ ] 保存済みデモを削除できる

### AC-P4: 品質基準

- [ ] `pnpm build` がエラーなく通る
- [ ] `pnpm lint` がエラーなく通る
- [ ] TypeScript のエラーが 0 件
- [ ] 生成コードの実行がサンドボックス内で行われる

## 検討事項・未決定事項

以下の項目は design.md で詳細を検討する。

### 1. LLM 連携方式

- **選択肢 A**: サーバーサイドで LLM API を呼び出す（Next.js API Routes / Server Actions）
- **選択肢 B**: クライアントサイドで直接 LLM API を呼び出す
- **検討ポイント**: API キーの管理、レスポンス速度、コスト、ストリーミング対応

### 2. コード実行のサンドボックス方式

- **選択肢 A**: iframe + Sandpack（CodeSandbox のエンベッド型サンドボックス）
- **選択肢 B**: eval ベースの動的レンダリング（セキュリティリスクあり）
- **選択肢 C**: Server Components でのサーバーサイドレンダリング
- **検討ポイント**: セキュリティ、パフォーマンス、ライブラリアクセス

### 3. データベース選定

- **選択肢 A**: SQLite（ファイルベース、セットアップ簡単、個人利用向き）
- **選択肢 B**: PostgreSQL（スケーラブル、チーム利用向き）
- **選択肢 C**: Turso（SQLite 互換のエッジ DB）
- **検討ポイント**: devcontainer / WSL 環境での動作、セットアップの簡便さ、将来的なスケーラビリティ

### 4. 生成コードの品質保証

- 生成されたコードがライブラリの規約（TypeScript strict、アクセシビリティ等）に準拠するか
- 存在しないコンポーネントを参照した場合のフォールバック処理
